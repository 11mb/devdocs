What is the Magento deployment configuration?

Magento's deployment configuration consists of the following files:

  • <your Magento install dir>/app/etc/config.php, which contains the list of modules
  • <your Magento install dir>/app/etc/env.php, which contains environment settings, such as:
    • Database credentials and connection settings
    • Cache storage settings
    • Enabled cache types
    • Your encryption key
    • Web routing parameters (base URLs, URL path to Magento Admin)
    • File system paths

Together, they are referred to as Magento's deployment configuration because they are created during installation and have information required to start Magento.

The deployment configuration replaces local.xml in earlier Magento versions. The deployment configuration contains a declarative array of configuration values.

Unlike other module configuration files, Magento's deployment configuration is loaded into memory when Magento initializes, is not merged with any other files (although config.php and env.php are merged with each other), and cannot be extended.

config.php and env.php contents

config.php and env.php are PHP files that return a multi-dimensional associative array.

On the first hierarchy level of this array are configuration segments. A segment has arbitrary content (a scalar value or a nested array) distinguished from each other by an arbitrary key—both the key and its value are defined by the Magento framework.

Magento\Framework\App\DeploymentConfig merely provides access to these sections but does not allow you to extend them.

On the next hierarchy level, items in each segment are ordered according to the module sequence definition, which is obtained by merging all modules’ configuration files, with the exception of disabled modules.

The following sections discusses the structure and contents of config.php and env.php.

  • config.php contents
  • env.php contents

config.php contents

Starting with build 0.74-beta10, config.php contains the list of modules only.

Disabled modules are not recognized by Magento; in other words, they don’t participate in merging configuration, in dependency injection, events, plug-ins, and so on. Disabled modules do not display in the storefront or Admin and don’t affect routing. The only practical difference of a module being disabled and being completely absent in the code base is that a disabled module is found by the autoloader, enabling its classes and constants to be reused in other code.

The value 1 or 0 indicates whether a module is enabled or disabled.

A snippet follows:

<?php
return array (
  'modules' =>
  array (
    'Magento_Core' => 1,
    'Magento_Store' => 1,
    'Magento_Theme' => 1,
    'Magento_Authorization' => 1,
    'Magento_Directory' => 1,
    'Magento_Backend' => 1,
    'Magento_Backup' => 1,
    'Magento_Eav' => 1,
    'Magento_Customer' => 1,
...
  ),
); ?>

env.php contents

env.php was introduced in build 0.74-beta10; before that build, config.php contained this data in addition to the list of enabled modules. The following table provides details about each env.php segment and its structure.

Segment Key Structure
Database db 
__/db
|__/connection
| |__/[default]
|   |-- host
|   |-- dbname
|   |-- username
|   |-- password
|   |-- model [mysql4]
|   |-- initStatements [SET NAMES utf8;]
|   |-- active [1]
|-- table_prefix
Resources resource 
__/resource
 |__/default_setup
   |-- connection [default]
Session storage session 
__/session
 |__/save
   |-- <files|db>
Admin URL path backend 
__/backend
 |-- frontName
Cache storage cache 
__/cache
 |__/frontend
   |__/<for future documentation>
Installation date install 
__/install
 |-- date
Encryption key encrypt 
__/crypt
 |-- key
Cache types cache_types 
__/cache_types
 |-- <enumerated cache types>

Updating to build 0.74-beta10

Magento 2 release 0.74-beta10 for the first time splits Magento's deployment configuration into two files:

  • <your Magento install dir>/app/etc/config.php
  • <your Magento install dir>/app/etc/env.php

As a result of this change, you must take action when upgrading from an earlier release. Your decision is basically do you want to keep your existing Magento 2 installation?

No, I do not want to keep my existing installation

Uninstall the Magento 2 software and reinstall it.

Yes, I want to keep my existing installation

To do this, you must manually update the deployment configuration as follows:

  1. Log in to your Magento server as a user with permissions to modify files in the Magento file system (for example, the web server user).
  2. Update the Magento software to release 0.74-beta10 or later.
  3. Change to the <your Magento install dir>/app/etc directory.
  4. Enter the following commands in the order shown:
    cp config.php env.php
vim env.php
  5. Delete the module list from env.php. The module list looks similar to the following:
    return array (
  'modules' =>
  array (
    'Magento_Store' => 1,
    'Magento_Directory' => 1,
    'Magento_Backend' => 1,
    'Magento_Backup' => 1,
    'Magento_Theme' => 1,
    'Magento_CacheInvalidate' => 1,
    'Magento_Eav' => 1,
    'Magento_Customer' => 1,
    'Magento_CatalogImportExport' => 1,
    'Magento_Indexer' => 1,
    'Magento_Cms' => 1,
    'Magento_Search' => 1,
    'Magento_Catalog' => 1,
   ... more
  ),
  6. Save your changes to env.php and exit the text editor.
  7. Open config.php in a text editor.
  8. Delete from config.php everything except the module list.
  9. Save your changes to config.php and exit the text editor.
